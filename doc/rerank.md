# Rerank 方案详解（面向 CloudAI 多知识库检索）

本文面向当前 CloudAI 的多知识库检索流程，系统说明 Rerank 的常见实现方案、适用场景、优缺点和落地路径。

## 1. 为什么需要 Rerank

当前检索结果在 [multi_retriever.go](file:///d:/Github/golang/src/CloudAI/internal/component/retriever/milvus/multi_retriever.go) 中先聚合，再按向量分数排序：

- 检索聚合逻辑位于 [multi_retriever.go:L35-L88](file:///d:/Github/golang/src/CloudAI/internal/component/retriever/milvus/multi_retriever.go#L35-L88)
- 仅按 `score` 排序位于 [multi_retriever.go:L90-L103](file:///d:/Github/golang/src/CloudAI/internal/component/retriever/milvus/multi_retriever.go#L90-L103)

这种方式简单高效，但会遇到典型问题：

- 语义相似但不回答问题的片段可能排在前面
- 多知识库合并后，跨库分数可比性不足
- 结果容易重复，浪费上下文窗口
- 业务优先级（时效、可信度、来源）无法体现

Rerank 的目标是：在召回候选集基础上，得到“更可回答问题”的最终 TopK。

## 2. 典型两阶段架构

推荐使用“两阶段检索”：

1. **召回阶段（Recall）**  
   向量检索（可叠加关键词检索）获取候选集，如 Top20~Top100。

2. **重排阶段（Rerank）**  
   对候选集重新打分排序，输出最终 TopK（如 Top3~Top10）给 LLM。

要点：

- 召回重在“找全”（高 Recall）
- 重排重在“找准”（高 Precision）

## 3. 常见 Rerank 方案

## 3.1 方案A：仅向量分数排序（当前方案）

### 思路

直接使用向量数据库返回的分数排序，取前 K。

### 优点

- 实现最简单
- 延迟最低
- 成本最低

### 缺点

- 相关性上限受限
- 不能利用关键词、业务信号
- 容易重复片段

### 适用场景

- 小规模、低成本、快速上线阶段
- 对答案质量要求一般的内部工具

## 3.2 方案B：融合重排（向量分 + 关键词分 + 业务分）

### 思路

把多个信号标准化后加权融合：

`final_score = w1*vector + w2*bm25 + w3*freshness + w4*source_priority`

其中：

- `vector`：语义相关性
- `bm25`：关键词匹配程度
- `freshness`：时效性（越新越高）
- `source_priority`：来源可信度/知识库权重

### 优点

- 性价比高，提升稳定
- 可解释性强，便于调参
- 工程可控，不依赖大模型服务

### 缺点

- 需要打通关键词检索
- 需要持续调参与监控

### 适用场景

- 大多数线上业务首选
- 对延迟和质量都敏感

## 3.3 方案C：Cross-Encoder 重排（主流高质量方案）

### 思路

对每个候选构造 `(query, chunk)` 输入 Cross-Encoder 模型，输出相关性分数，再排序。

### 优点

- 相关性提升明显
- 对复杂问题、长 query 更稳

### 缺点

- 成本和延迟较高
- 候选数越多，耗时越高

### 适用场景

- 对答案质量要求高
- 候选数可控（通常 20~80）

## 3.4 方案D：MMR 多样性重排（去重增强）

### 思路

在高分候选中加入“多样性惩罚”，避免重复内容占满 TopK。

目标函数可理解为：

- 既要和 query 相关
- 又要与已选文档保持差异

### 优点

- 显著减少重复片段
- 提升上下文信息密度

### 缺点

- 仅靠 MMR 不能替代语义精排
- 参数不当会引入无关片段

### 适用场景

- 文档切片重复较多
- 多知识库聚合后重复率高

## 3.5 方案E：LLM Judge 重排

### 思路

让 LLM 对候选进行“是否能回答问题”的判别打分，再排序。

### 优点

- 语义判断能力强
- 对复杂问答效果可观

### 缺点

- 成本高、延迟高
- 稳定性受模型与提示词影响

### 适用场景

- 小流量高价值场景
- 对最终质量极敏感

## 4. 推荐实践（按阶段落地）

## 4.1 Phase 1：低成本上线

- 保持现有向量召回流程
- 增加基础过滤：最小分数阈值、空内容过滤
- 引入简单去重（标题/文档ID/近重复）

## 4.2 Phase 2：工业可用

- 引入混合信号融合（方案B）
- 增加 MMR 去重（方案D）
- 建立离线评测集，按指标迭代权重

## 4.3 Phase 3：质量优先

- 在候选集上引入 Cross-Encoder（方案C）
- 对极高价值请求按策略启用 LLM Judge（方案E）
- 建立分级策略：普通请求走轻量重排，关键请求走强重排

## 5. 多知识库场景下的关键注意点

结合 [multi_retriever.go](file:///d:/Github/golang/src/CloudAI/internal/component/retriever/milvus/multi_retriever.go)：

- 各知识库向量分数可能分布不同，建议做分库归一化后再融合
- 每个 KB 的 `TopK` 不宜过大，避免放大尾部噪声
- 对 KB 设置权重（权威库、业务主库）常常有效
- 建议对候选设置总预算（如全局最多 50 条）控制重排成本

## 6. 指标与评测建议

上线前后建议持续观察：

- 检索指标：Recall@K、MRR、nDCG
- 生成质量：答案准确率、引用命中率、人工评审通过率
- 系统指标：P95/P99 延迟、Rerank 成本、超时率

没有评测闭环，Rerank 很容易“看起来在优化，实际在漂移”。

## 7. 一套可执行的默认配置建议

可作为 CloudAI 的初始默认：

- 召回：每库 Top3~Top5，合并后最多 40 条
- 预过滤：向量分阈值 + 基础去重
- 重排：融合分（向量 + 关键词 + 时效 + 来源权重）
- 去重：MMR 或近重复抑制
- 输出：最终 Top5 给 LLM

当流量与质量要求提升时，再叠加 Cross-Encoder。

## 8. 总结

在实际工程中，Rerank 的最佳路径通常不是“一步到位上最重模型”，而是：

- 先有可用的召回和排序
- 再用融合分与去重提升稳定性
- 最后按业务价值引入高成本高质量重排

对当前 CloudAI，多知识库场景最推荐的路线是：  
**向量召回 + 融合重排 + 去重（MMR）**，再按需要逐步引入 Cross-Encoder。
